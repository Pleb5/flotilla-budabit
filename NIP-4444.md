# NIP-4444: Minimal Inbox-Relay Direct Messages

`draft` `optional`

## Abstract

This NIP defines a minimal direct-message profile used in this project:

- DM events use kind `4444`
- messages are encrypted with NIP-44
- events carry only one required metadata tag: the recipient `p` tag
- delivery uses recipient-published messaging inbox relays (kind `10050` list)

It is intentionally simple and does not use gift wrap, seals, rumors, or threaded DM metadata.

## Motivation

The goal is a low-complexity DM format that is easy to implement, interoperable, and reduces spam by routing messages to relays explicitly declared for messaging.

## Terms

- **Sender**: event author (`pubkey` field)
- **Recipient**: pubkey in the first `p` tag
- **Inbox relay list**: recipient's kind `10050` relay list

## Event Kind

- DM event kind: `4444`

## Message Event Format

A DM is a normal signed Nostr event with:

- `kind: 4444`
- `content: <NIP-44 ciphertext>`
- `tags: [["p", "<recipient-hex-pubkey>"]]`

### Tag Rules

- A sender MUST include a `p` tag for the recipient.
- A sender SHOULD include exactly one `p` tag.
- Clients implementing this profile SHOULD NOT add other tags for DM metadata (`e`, `a`, `subject`, etc.).

This profile treats DMs as standalone messages, not threaded events.

## Encryption

- Encrypt plaintext with NIP-44 using sender secret key and recipient public key.
- Put the ciphertext directly in `content`.

No wrapper event format is used in this profile.

## Recipient Relay Discovery

To send, the client loads the recipient's messaging relay list event (kind `10050`) and extracts relay URLs from relay tags.

- Preferred tag form: `relay` tags, e.g. `["relay", "wss://relay.example"]`
- Clients MAY accept legacy/alternate relay tag parsing if their stack already supports it.
- Clients SHOULD normalize relay URLs, drop invalid URLs, and de-duplicate.

## Sending Rules

1. Resolve recipient inbox relays from recipient kind `10050` list.
2. Encrypt plaintext with NIP-44 to recipient pubkey.
3. Create kind `4444` event with only recipient `p` tag.
4. Publish to recipient inbox relays.

Implementation note from this project: compose is blocked unless both sender and recipient have at least one inbox relay configured.

## Receiving and Sync Rules

For a local user pubkey `U`, subscribe on `U`'s inbox relays using both filters:

- incoming: `{ "kinds": [4444], "#p": [U] }`
- outgoing copy: `{ "kinds": [4444], "authors": [U] }`

This captures messages sent to the user and messages authored by the user.

Recommended client behavior is to do both:

- bounded backfill (this project uses ~2 weeks)
- live subscription

## Decryption Rules

To decrypt an event for local user pubkey `U`:

- if `event.pubkey == U`, decrypt using recipient pubkey from first `p` tag
- otherwise, decrypt using `event.pubkey` (the sender)

If decryption fails, clients SHOULD treat the payload as undecryptable and avoid crashing.

## Conversation Model

This profile is one-to-one by default.

- Clients SHOULD send only one recipient `p` tag.
- Group-style DM composition is out of scope for this profile.

## Privacy and Spam Notes

- Relay operators can still see sender, recipient tag, and timing metadata.
- Ciphertext content is protected by NIP-44.
- Inbox relay routing helps reduce unsolicited delivery to unrelated relays.

## Example DM Event

```json
{
  "kind": 4444,
  "content": "<nip44-ciphertext>",
  "tags": [["p", "<recipient-hex-pubkey>"]],
  "created_at": 1710000000,
  "pubkey": "<sender-hex-pubkey>",
  "id": "<event-id>",
  "sig": "<signature>"
}
```

## Example Messaging Relay List (Recipient)

```json
{
  "kind": 10050,
  "tags": [
    ["relay", "wss://inbox1.example"],
    ["relay", "wss://inbox2.example"]
  ],
  "content": "",
  "pubkey": "<recipient-hex-pubkey>",
  "id": "<event-id>",
  "sig": "<signature>"
}
```
